# 授权管理 — 定义

> **状态**：需求规划阶段，等功能基本完成后再设计实现。
> 当前只定义方向和框架预留，不展开具体功能划分。

---

## 一、商业模式

**订阅制**，按功能深度划分三档 + 免费：

| 档位 | 名称 | 定位 |
|------|------|------|
| **Tier 0** | Free | 核心体验，所有 View 的基础能力 |
| **Tier 1** | Pro | 高级能力（AI、标注、Thought、协同等） |
| **Tier 2** | Premium | 专业能力（GraphView、云同步、多设备、高级导出等） |

---

## 二、划分原则

**按功能深度划分**（维度 B）：

- 所有 View 的基础形态对所有用户开放（编辑、阅读、浏览）
- 深度功能按档位解锁
- 具体的功能边界待功能矩阵稳定后确定，可能包括：
  - 部分 Block 功能限制
  - 使用次数限制
  - Workspace 数量限制
  - 高级 View/variant 限制

---

## 三、框架预留

### 3.1 授权级别类型

```typescript
type LicenseTier = 'free' | 'pro' | 'premium';
```

### 3.2 插件级授权

插件注册时声明最低授权要求：

```typescript
interface PluginRegistration {
  // ...已有字段
  tier: LicenseTier;    // 整个插件的最低授权级别
}
```

### 3.3 功能级授权

同一插件内的不同功能可以有不同授权要求（粒度更细）：

```typescript
interface FeatureRegistration {
  id: string;            // 如 'pdf.highlight', 'note.thought'
  tier: LicenseTier;     // 该功能的最低授权级别
  pluginId: string;      // 属于哪个插件
}
```

### 3.4 框架行为

```
框架启动
  → 读取用户授权级别
  → 插件注册时检查 tier
    → 用户授权 ≥ 插件 tier → 正常激活
    → 用户授权 < 插件 tier → 不激活 / 降级 / 提示升级
  → 功能调用时检查 FeatureRegistration.tier
    → 授权不足 → 拦截并提示升级
```

## 四、实施时机

1. **当前**：框架代码中预留 `tier` 字段，但不实现检查逻辑
2. **功能基本完成后**：确定具体的功能→档位映射表
3. **发布前**：实现完整的订阅系统（在线验证、到期管理、付费墙 UI）

---

## 五、约束

1. **Free 必须可用**：Free 用户能完整体验核心编辑、阅读、浏览，不能连基础功能都锁住
2. **注册制**：授权要求由插件/功能通过注册声明，框架不硬编码哪些功能属于哪个档位
3. **优雅降级**：授权不足时，功能可见但禁用（带升级提示），不是完全隐藏
