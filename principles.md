# KRIG Note — 重构原则

> 本文档记录从 mirro-desktop 实验中沉淀的设计原则。
> 等功能矩阵稳定后，基于这些原则进行高度抽象的重构。

---

## 一、分层设计

分层是架构的骨架。每一层有明确的职责边界和依赖方向。

### 原则

1. **单向依赖**：上层依赖下层，下层永远不知道上层的存在
2. **层间契约**：层与层之间通过接口通信，不直接访问对方的内部实现
3. **可替换性**：任何一层的实现可以替换，只要满足接口契约（比如 ProseMirror 换成其他编辑器引擎，上层不需要改）

### mirro-desktop 当前分层（待审视）

```
Electron Main（进程管理、窗口、系统 API）
    ↓
Module Layer（Browser、Note、PDF 等业务模块）
    ↓
Editor Engine（ProseMirror）
    ↓
Node Layer（Block / Container 的 View + Commands + Plugin）
```

重构时需要重新审视：每层职责是否清晰？是否有跨层调用？依赖方向是否严格单向？

## 二、抽象层次

### 原则

1. **相似即可抽象**：当两个以上功能有相同的结构或行为模式时，提取共同抽象
2. **抽象要有收益**：抽象的目的是减少重复、统一行为、降低认知负担——不是为了抽象而抽象
3. **先具体后抽象**：在 mirro-desktop 中先用具体实现验证，确认模式稳定后再提取到抽象层

### 已识别的抽象机会

（随开发推进持续补充）

- **EmbedBlock 模式**：Video、Tweet、NoteLink 等都是"外部资源嵌入"，结构相似（URL/ID + 元数据 + 预览视图）
- **Converter 模式**：所有 Block 都需要 Atom ↔ ProseMirror 双向转换，接口统一
- **Container 模式**：所有容器共享折叠/展开、子节点管理、嵌套等行为

---

## 三、Block + Container 二元模型

这是编辑器的核心模型，在 mirro-desktop 中已验证有效。

### 原则

1. **只有两种东西**：Block（叶子）和 Container（容器）。Container 继承 Block
2. **能力不丢失**：Block 放入任何 Container 中，能力完全保留
3. **整体移动**：Container 移动时，容器 + 全部子节点一起移动
4. **缩进即包含**：视觉缩进决定包含关系

### 重构时的进化方向

- 当前每种 Block 各自实现 View + Commands，重构时考虑基于 EmbedBlock 等抽象基类减少重复
- Container 的共享行为（fold/unfold、addChild、dissolve）应该有更统一的实现

---

## 四、模块自包含

### 原则

1. **一类节点 = 一个模块**：View、Commands、Plugin 聚合在一起，不分散到全局
2. **模块导出契约**：每个模块通过 index.ts 导出标准接口
3. **注册即生效**：新增节点只需要实现模块 + 注册，不需要修改框架代码

### 重构时的进化方向

- mirro-desktop 中注册分散在多个文件（schema、NoteApp、SlashMenu、converters...），重构时考虑声明式注册（一个配置搞定）

---

## 五、框架与插件分离

这是最高层次的架构原则——决定了什么是稳定的骨架，什么是可迭代的扩展。

### 原则

1. **面向用户的功能都是插件**：框架只提供组织和通信的骨架，所有具体功能（Block 类型、View 类型、AI 能力、导入导出格式等）都以插件形式实现
2. **框架稳定后，开发在插件层**：框架一旦成熟，日常开发和调试集中在插件级别，不需要碰框架代码
3. **自顶向下的 UI 框架层次**：框架由 4 层组成——

```
macOS Application（Application Menu）
  └─ Window (Shell)
       └─ Workspace
            ├─ NavSide（+ 自己的 Overlays）
            └─ Slot（+ 自己的 Overlays）
                 └─ View + Toolbar（+ 自己的 Overlays）
```

| 层 | 职责 |
|---|---|
| **macOS Application** | 应用生命周期 + Application Menu（注册机制） |
| **Window (Shell)** | 窗口骨架 + 全局共享 UI（WorkspaceBar、全局 Overlays） |
| **Workspace** | 独立工作环境，每个 Tab 一个，驱动 NavSide 和 Slot 的内容 |
| **NavSide + Slot** | NavSide = 导航操作；Slot = 内容渲染，每个 Slot 放一个 View |

4. **Workspace 是核心调度单元**：Workspace 是逻辑实体，不是某个 UI 区域。它驱动 NavSide 显示什么内容、Slot 显示什么布局，拥有自己的状态（模式、布局比例等）。WorkspaceBar 只是管理 Workspace 生命周期的 UI 控件（创建、切换、关闭）
5. **View 是插件的载体**：框架定义 View 接口（生命周期、Toolbar 管理、Slot 通信），插件实现具体的 View（NoteView、PDFView、WebView 等）。每个 View 拥有自己的 Toolbar，不由外部统一管理
6. **NavSide 操作一致性**：交互模式（布局、导航方式）由框架统一定义，保证用户操作体验一致；具体操作内容由插件通过接口注册。NavSide 是共享的渲染容器，内容由当前活跃 Workspace 驱动
7. **Overlay 属于子视图**：Overlay（浮窗）不是独立的一层，而是每个子视图自己的能力。NavSide、Slot、View、Toolbar 各自管理自己的浮窗（右键菜单、下拉菜单、SlashMenu 等）
8. **Slot 间通信**：Slot 之间通过协议通信，框架只做消息路由，不理解消息内容。这是贯穿各层的通信机制，不是独立的层

## 六、命名与可描述性

### 原则

1. **产品命名**：应用统一命名为 **KRIG-Note**
2. **万物皆可命名**：每一个层次、每一个元素都必须有标准化的命名，并单独列表解释其含义。这包括框架的每一层、每一个类、每一个接口、每一个协议、每一个函数。如果一个概念解释不清楚，说明它不具备可实施性——命名即设计，命名模糊则设计模糊
3. **用产品描述产品（自举验证）**：使用 KRIG-Note 自身来描述 KRIG-Note 的架构和设计。如果用自己的产品都无法清晰地组织和表达自己的设计文档，说明产品的表达能力不足，设计或开发存在问题。这是最真实的 dogfooding——产品既是工具，也是自己的第一个用户

## 七、存储与交换格式

### 原则

1. **存储格式对人和 AI 都友好**：存储格式不应只服务于编辑器引擎，还应便于外部工具（AI、脚本、版本管理）直接读写
2. **Markdown 作为通用交换格式**：Markdown 是人、AI、版本控制系统都能无障碍读写的格式。设计文档、知识库内容应支持 Markdown 作为导入导出的通用桥梁
3. **自举验证的双轨制**：Markdown 作为设计文档的唯一真相源（AI 读写），NoteView 作为展示验证（人在 App 里阅读，验证产品的表达能力）。两者互补，缺一不可

### 发现（来自实践）

ProseMirror Doc JSON 嵌套深、marks 混在结构里，AI 可以读但体验极差。这说明：**如果未来 NoteView 支持 Markdown 原生读写（存储即 .md），设计文档的源文件和产品展示就能合一，自举验证的成本降为零。**

## 八、UI 表现层配置化

### 原则

1. **视觉属性不硬编码**：icon、button 颜色、bar 高度、间距、字体大小、圆角等视觉属性不散落在代码中，统一通过配置文件（主题文件）定义
2. **表现与逻辑分离**：代码只负责结构和交互逻辑，视觉风格由配置驱动。UI 工程师修改配置文件即可调整视觉，不需要碰业务代码
3. **单一视觉源头**：所有同类元素（如所有 primary button、所有 toolbar）的视觉属性从同一个定义派生，保证全局一致性
4. **主题可切换**：暗色/亮色/自定义主题只需切换配置文件，不需要修改组件代码
