# KRIG 设计哲学与产品原则

> 文档类型：产品设计哲学
> 产品名称：KRIG / KRIG Note
> 状态：持续更新 | 创建日期：2026-03-26 | 版本：v1.1
>
> **文档目的**：记录 KRIG 在设计决策过程中形成的核心哲学原则。
> 这些原则不是功能规格，而是**判断标准**——当模块设计出现方向性选择时，以此文档为第一参照。

---

## 一、核心产品哲学

### 原则 1：KRIG 不替代思考，而是组织思考

> **"KRIG 不替代思考，而是组织思考。"**

这是 KRIG 区别于当前所有 AI 工具的根本立场。

当前大多数 AI 工具的逻辑是：替用户得出结论，让用户验收结果。KRIG 的逻辑相反——系统帮助用户看清自己思考的结构，结论永远属于用户。

**三层含义：**

| 层次 | 含义 |
|------|------|
| 尊重 | 用户的结论是用户的，不是系统的 |
| 赋能 | 系统让用户看到自己思考的结构 |
| 对话 | 不同用户的思考结构可以相遇、比较 |

**对各模块的影响：**

| 模块 | 替代思考的错误做法 | 组织思考的正确做法 |
|------|-------------------|-------------------|
| PDF 提取 | 自动生成摘要结论 | 呈现结构，用户决定权重 |
| 知识图谱 | 自动建立所有连接 | 展示可能的连接，用户确认 |
| AI 分屏 | AI 直接回答问题 | AI 帮用户看清问题的边界 |
| Note | 自动整理笔记 | 保留用户的原始思路痕迹 |

每个模块设计都面临一个岔路口：系统多做一点，还是用户多做一点。此原则是判断标准。

---

### 原则 2：知识视图是个人的，不应被系统统一

> **"知识的乐趣就在不同的人会有不同的思考，产生不同的视图。如果都一致了，就失去了思考的乐趣。"**

不同的人用同一批知识材料，会激活不同的节点，建立不同的连接。这种差异不是噪音，而是知识生长的机制本身。

科学史上最重要的突破，往往不是发现了新事实，而是用新的视图重新组织了已知的事实。

**两层结构：**

```
知识的内容   →  可以趋向共识（事实、数据、引用）
知识的视图   →  应该保持多样（解读、连接、权重）
```

**对知识图谱模块的影响：**

- 知识节点（Block）：共享的基础层
- 节点之间的连接和权重：个人视图层
- 不同用户的图可以叠加、比较、对话，但不强制统一

## 二、Block 的设计哲学

### 原则 3：Block 是数据组织单元，不只是视图渲染单元

在普通笔记工具（Notion、Obsidian）里，Block 是视图渲染单元——它的职责是"怎么显示"。

在 KRIG 里，Block 是**数据组织单元**——它同时具有双重身份：

| 身份 | 职责 | 层次 |
|------|------|------|
| 视图侧 | 交给渲染器输出可视化内容 | 表现层 |
| 数据侧 | 作为知识图谱里的实体节点，携带语义、可被引用、可建立关系 | 数据层 |

**含义：**

一个 LaTeX 公式 Block，不只是"渲染出一个公式"。它在知识图谱里是一个概念节点，可以和其他 Block（论文、实验数据、笔记）建立语义关联。

**这是 KRIG 和 Obsidian / Notion 的根本分叉点：**

- Notion / Obsidian 的 Block 止步于视图层，数据组织依赖文件系统或数据库表
- KRIG 的 Block 从设计之初就是 SurrealDB 的一等公民——每个 Block 记录天然是知识图谱的潜在节点
- Module 3（Knowledge Graph）不是一个独立功能，而是 Block 数据层的自然呈现

---

## 三、数据库与视图的设计哲学

### 原则 4：视图是数据自然形态的反映，不是用户手动选择的结果

> **KRIG 不提供手动选择的数据库视图。视图是数据自然形态的反映，由系统根据 Block 的语义关系和数量自动推断呈现，用户无需提前规划数据结构。**

**Notion 的问题所在：**

Notion 的数据库视图是手动的、显式的——用户要先建数据库，再手动选择视图类型。这把数据组织的认知负担完全压在用户身上，也是为什么大多数人用了 Notion 很久，都没真正用过它的数据库功能。

**Notion 的范式（从结构到视图）：**
```
用户手动建数据库 → 用户手动选视图类型 → 系统展示
```

**KRIG 的范式（从内容到结构）：**
```
用户输入想法（Block）→ SurrealDB 自动结构化 → 系统推断最适合的视图 → 自动呈现
```

**对 Module 3 和 SurrealDB Schema 设计的影响：**

- 不需要实现 18 种用户手动选择的数据库视图
- 需要实现：根据 Block 的类型、数量、语义关系，自动判断最合适的呈现形式
- 当同类节点积累到一定数量，系统可以自动识别"这批数据适合用列表 / 时间线 / 关系图呈现"

## 四、交互设计哲学

### 原则 5：Tab 组件是阅读思考流程的容器，不只是布局工具

Notion 的选项卡（Tab）是纯粹的视觉布局组件——在同一区域切换不同内容，节省页面空间，各 Tab 之间没有语义关系。

KRIG 的 Tab 有更深的含义：它是**一个完整阅读思考流程的容器**。

**标准 Tab 结构：**

```
原文  |  译文  |  批注  |  Thought
```

用户在同一个 Block 内完成"看原文 → 看译文 → 记批注 → 写自己的想法"的完整流程，不需要跳出去开新 Block 或切换页面。这和 KRIG 的重力中心原则一致——思考发生在哪里，工具就在哪里，不打断心流。

**Thought Tab 的特殊价值：**

Thought 作为 Tab 存在，意味着它天然和这个 Block 绑定了语义关系——这段思考是针对这个具体 Block 的，而不是游离的文字。存入 SurrealDB 时，这个关系自动成立：

```
Block(原文节点) ──has_thought──> Thought(用户思考)
```

不需要用户手动建立连接，知识图谱里的边自然产生。这比 Notion 的 Tab 深得多——Notion 的 Tab 只是视觉布局，切换的内容之间没有任何语义关系。

**对 Module 2（Smart Editor）的影响：**

- Tab 组件不是通用布局工具，而是为"阅读 + 思考"场景专门设计的
- Thought Tab 是标准配置，不是可选项
- Tab 内容的切换不影响 Block 在图谱中的节点身份

## 五、知识图谱构建哲学

### 原则 6：关系是懒惰构建的，不是急切构建的

> **"永远不做全图查询，永远不做同步关系推断，永远只加载当前需要的局部数据。"**

**背景：**

一个重度用户积累 100 本书 + 1000 篇笔记，可能产生 350 万个节点、1000–3500 万条边。性能问题不在于存储，而在于在错误的时机查询了过大的范围。

核心原则：关系是 **Lazy（懒惰）** 构建的，不是 **Eager（急切）** 构建的。

---

**三个触发原则：**

**1. 局部触发，不全局扫描**

```
用户打开一个 Block    →  只构建这个 Block 的直接关系
用户搜索一个概念      →  只扩展和这个概念相关的子图
用户手动连接两个节点  →  只更新这两个节点之间的边
```

不存在"重新扫描所有节点"这个操作。

**2. 时机触发，不写入即触发**

```
Atom 写入 SurrealDB    →  只存储，不构建关系（导入阶段）
用户第一次访问该 Block →  触发关系推断（使用阶段）
后台空闲时             →  渐进式处理积压的待推断队列
```

关系推断和数据写入是解耦的两个独立阶段。

**3. 增量更新，不全量重建**

```
用户修改一个 Block  →  只更新这个 Block 涉及的边
每条边有自己的 dirty 标记，只有被标记的才重新计算
不存在"重建整个图谱"这个操作
```

---

**冷热数据分层（配合关系懒构建）：**

```
热数据（实时访问）：当前打开的 Note 及其直接关联节点
                   → 常驻内存，毫秒级响应

温数据（近期访问）：最近活跃的节点
                   → SurrealDB 本地查询，百毫秒级

冷数据（历史归档）：长期未访问的书本内容
                   → 只存储，不主动加载到图谱视图
```

**对 Module 3 实现的影响：**

- 导入一本书时，用户看到的是"正在后台建立知识关联"的进度提示，界面不阻塞
- 图谱视图永远只渲染当前焦点节点的局部子图，用户移动焦点时动态加载
- 批量写入使用 batch insert（几千条一批），不逐条事务

---

### 原则 7：知识图谱节点由系统提议，用户确认

> **"系统是侦察员，用户是决策者。"**

词频统计、关键词提取、命名实体识别等手段产生的概念候选，是系统对文本的初步侦察结果。这些候选进入 `inference_queue` 暂存，**不自动写入正式图谱**。

节点的"存在权"由用户决定。

**系统可以做的：**
- 生成概念候选节点
- 标注置信度
- 推荐潜在关系

**系统不可以做的：**
- 未经用户确认将候选节点写入 `concept` 表
- 将系统推断的关系和用户确认的关系混合存储

**和 P1 的关系：**

P1 确立了"组织思考而非替代思考"的全局原则，P7 是 P1 在知识图谱节点生命周期管理上的具体落地——确认动作是用户对自己知识结构的主动声明，不能被系统绕过。

**对 Module 3 的影响：**

| 阶段 | 系统行为 | 存储位置 |
|------|----------|----------|
| 文本分析完成 | 生成候选节点 + 置信度 | `inference_queue` |
| 用户确认 | 节点正式进入图谱 | `concept` 表 |
| 用户忽略 / 过期 | 候选节点自动清理 | — |

**词频在此原则中的定位：**

词频统计是候选节点生成链路的**第一个漏斗**——从海量文本信号中粗筛出高频概念候选，再经由关键词提取、命名实体识别逐层过滤，最终以候选形式呈现给用户确认。词频本身不直接产生节点，而是驱动后续链路的原始信号。

```
词频统计（第一漏斗，量大噪音多）
    ↓
关键词提取 YAKE/TextRank（第二漏斗，语义权重）
    ↓
命名实体识别 NER（第三漏斗，人/地/概念/术语）
    ↓
inference_queue（候选暂存，等待用户确认）
    ↓
concept 表（用户确认后正式写入）
    ↓
Module 4 图谱渲染
```
