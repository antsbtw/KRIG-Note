> **作用域**：KRIG graph 子系统从"算法画的"提升到"专业编辑器审美"的整体路线
> **触发**：用户对比 G6 生态树后明确不满"图的组织方式"
> **关联**：[KRIG-Graph-Canvas-Spec.md](./KRIG-Graph-Canvas-Spec.md) §6 phase 之上的横切计划

> 视觉问题先忍一忍，等编辑闭环完整再统一调。

## §0 触发与定位

### §0.1 用户反馈原文

> "我觉得我们的图确是很不美观，请观察一张 G6 的生态树图，边和点都很优美，
> 而我们的是很难看的。难道这就是 ELK 的极限吗？"

> "我对我们的图的组织方式是不满意的。"

### §0.2 关键澄清：不是 ELK 极限

ELK 完全能输出 G6 那样的效果。当前丑是**渲染层 + 默认值 + 视觉编排**没做到位，跟布局算法本身无关。

### §0.3 不满分类（4 维，待用户多选确认）

| 维度 | 当前状态 | 是否用户不满 |
|---|---|---|
| **A. 视觉美观** | 节点 / 边 / 配色都不够精致 | ⏸ 待确认 |
| **B. 布局组织** | mrtree 默认紧凑直线，"算法画的"无设计感 | ⏸ 待确认 |
| **C. label 处理** | label 跟节点分离，箭头能穿过 label 文字 | ⏸ 待确认 |
| **D. 整体编排** | 多根树 / 孤立节点处理粗糙，画布利用率低 | ⏸ 待确认 |

**用户首次回答时勾选**，后续按结果调整阶段优先级。

### §0.4 跟 Canvas Spec 的关系

Canvas Spec §6 列的是**功能 phase**（B4.2.a 编辑、B4.3 凝结、…）。
本文档列的是**视觉抛光阶段**，属于横切关注（cross-cutting），跟功能 phase 并行。

按当前判断（见 §6.1），**视觉抛光优先于 B4.2.b**。

## §1 整体节奏

```
阶段 1：诊断 + 基线（半天）        — 不动代码，量化"现在多丑"
   ↓
阶段 2：边渲染重构（1-2 天）      — 贝塞尔曲线 + 端点 clipping + 4 选项真正可视化
   ↓
阶段 3：节点 + label 一体化（1 天）— label 紧贴节点 / 跟方向走 / 不被穿透
   ↓
阶段 4：默认配色 + 留白系统（半天）— G6 同款审美底子
   ↓
阶段 5：高级编排能力（1 天）       — 子树折叠 / 智能间距 / 多根对齐 / LOD
```

**总投入 3-5 天**，分散在多次会话。每阶段独立 commit，做完用户验收。

**核心原则**：每阶段有"体验升级一档"的可见结果，不行就停下调整再继续。

### §2.1 目的

### §2.2 产出

`docs/graph/Visual-Audit-2026-04.md`，包含：

3. **跟 G6 / Cytoscape / Mermaid 三个参考做并排对比**：

### §2.3 不动代码的理由

避免抹脑袋修 → 修了不重要的、漏了关键的。**量化才能对症**。

## §3 阶段 2：边渲染重构

### §3.1 核心改动

让 `layout.edge-style` 4 个选项**真正在视觉上不同**。

| 选项 | 现在画什么 | 应该画什么 |
|---|---|---|
| 直线 | 直线段（OK）| 直线段（保持）|
| 直角 | 直角折线（OK，但端点穿 label）| 直角折线 + 端点 clip 到节点边缘 |
| 折线 | 同直角（**bug**）| 多段折线，跟直角不同 |
| 曲线 | 同折线（**bug**）| 贝塞尔曲线，平滑 |

### §3.2 实施清单

1. **新增 `BezierCurveShape.ts`** 或扩展 `LineSegmentShape.ts`：
   - 接收 ELK 输出的 `bendPoints`
   - 用 Three.js `CatmullRomCurve3` / `QuadraticBezierCurve3` 生成平滑曲线
   - 渲染为 `MeshLine`（粗细可控，比原生 Line 好看）
2. **改 projection（'tree' projection）的 customizeLine**：
   - 根据 `layout.edge-style` 选不同 shape
   - splines / polyline / orthogonal / straight 各走各的
3. **端点 clipping 升级**：
   - 当前 `clipLineEndpointsToShapes` 用 mesh 的 Box3
   - 加上 **label bbox 也参与求交**（避免穿 label）
   - 给端点留 padding（如 4px）让箭头不贴节点边缘
4. **箭头视觉升级**：
   - 当前箭头三角形偏粗
   - 改用更细更锐利的箭头形状

### §3.3 用户可见

边变美。

## §4 阶段 3：节点 + label 一体化

### §4.1 核心改动

解决"label 跟节点分离 → 边穿过 label"的根本问题。

**思路对齐 G6**：节点 + label 视为**一个视觉单元**，参与布局时算"含 label 的总 bbox"，渲染时 label 紧贴节点。

### §4.2 实施清单

1. **label-aware sizing 真正生效**：
   - B3.4.5 已有 measureLabel，但 mrtree 当前没消费 label bbox 撑大节点
   - 让所有 layout 算法都用 `getInstanceBoxSize(substance, labelBbox)` 算总 bbox
2. **labelLayout 跟方向走**：
   - 现在 substance.visual.labelLayout 固定（`below-center`）
   - 按 layout direction 自动选：
     - DOWN/UP → label 在节点下方/上方
     - LEFT/RIGHT → label 在节点右侧/左侧
   - 用户可在 Inspector 节点 Tab 覆盖（B4.2.b 实施时一起）
3. **label 字体优化**：
   - 当前 SVG label 渲染可能字重不一致
   - 调整 font-weight / font-family / 字间距
4. **节点 hover 状态**：
   - 当前只有 selected 高亮
   - 加 hover 高亮（橙色描边 `HIGHLIGHT_HOVER`）

### §4.3 用户可见

节点和 label 像一个整体，不再有"穿透感"。

## §5 阶段 4：默认配色 + 留白系统

### §5.1 核心改动

建立审美底子，不再"算法画的"。

### §5.2 实施清单

1. **新增 `theme.ts`**：
   - 集中定义 default fill / border / text / edge colors
   - 提供 dark / light 两套（v1 只用 dark，留接口）
   - 借鉴 G6 / Figma 的配色（深蓝主色 + 灰色辅助）
2. **重新调整 substance defaults**：
   - 当前 krig-concept 等内置 substance 的颜色是各自硬编码
   - 改为引用 theme（保留 substance 自定义优先级）
3. **fitToContent padding 调优**：
   - 当前 padding 不知道多少
   - 加 15-20% 留白（视图边缘到节点 bbox 的距离）
4. **节点最小尺寸 + 间距默认值**：
   - mrtree 默认 60 太挤
   - 调到 80（spacing.node）+ 100（spacing.layer）
   - Inspector 控件值跟着改

### §5.3 用户可见

默认打开就好看，不需要调任何参数。

## §6 阶段 5：高级编排能力

### §6.1 核心改动

从"画一棵树"升级到"组织信息建筑"。

### §6.2 实施清单

1. **多根树智能对齐**：
   - 当前 KRIG-Note-Concept.md 有 1 个 contains 真根 + 16 个孤立点（挤一团）
   - 解决：孤立节点单独走 grid 子布局，contains 树主导主区域
2. **子树折叠/展开 hint**：
   - 节点有 ≥3 个子时，加"折叠"图标
   - 折叠后子树缩为单节点（用 ELK group 机制）
   - 注：Canvas Spec §7 长期演进里有此项，但视觉提升刚需
3. **背景网格 / 标尺（可选）**：
   - 像 Figma 一样浅色网格背景
   - 用户拖节点时有对齐感
4. **缩放层次细节（LOD）**：
   - 缩小到一定阈值时，hide label（避免遮挡）
   - 进一步缩小，节点缩成小点（看整体结构）

### §6.3 用户可见

图谱有"专业工具"感。
