# KRIG Note — 原则提炼草稿（待讨论）

> 从 mirro-desktop 实验中提炼出的所有设计原则。
> 标记 ✅ = 建议保留，❓ = 需要讨论，❌ = 建议不带入重构。
> 讨论确认后，有价值的原则合并到 principles.md。

---

## 一、分层设计原则

### 1.1 单向依赖 ✅
上层依赖下层，下层永远不知道上层的存在。

### 1.2 层间契约 ✅
层与层之间通过接口通信，不直接访问对方的内部实现。

### 1.3 可替换性 ✅
任何一层的实现可以替换，只要满足接口契约。比如编辑器引擎从 ProseMirror 换成别的，上层业务不需要改。

### 1.4 分层不可侵入 ✅
上层只能调用/包装下层，不能修改下层已验证的行为。Container 层是 Block 层的上层，只能包装，不能侵入。

### 1.5 三层数据架构 ❓
Atom（存储层，框架无关） → View（配置层，渲染元数据） → Block（运行时，UI 实例）。
> 问题：View 层在 mirro-desktop 中是否充分验证？重构时是否沿用？

## 二、核心模型原则

### 2.1 Block + Container 二元模型 ✅
编辑器只有两种对象：Block（叶子）和 Container（容器，继承 Block）。

### 2.2 两个维度思考 ✅
任何功能只需回答两个问题：视图怎么渲染？操作怎么交互？

### 2.3 能力保留不变量 ✅
Block 放入任何 Container 中，视图和操作能力完全保留，不可丢失。Container 嵌入另一 Container 同理。

### 2.4 整体移动不变量 ✅
Container 移动时，容器 + 全部子节点一起移动，不可拆解。

### 2.5 内容合法性不变量 ✅
每个 Container 的 content 表达式定义了合法内容，任何操作都不可破坏。

### 2.6 缩进即包含 ✅
视觉缩进决定包含关系。容器标题行没有缩进，行为上等同普通 Block。

### 2.7 操作全局一致性 ✅
同一个操作（如"转为 Heading"）无论从哪个入口调用（菜单、快捷键、API），行为完全一致。一个操作 = 一个函数，不允许多个实现。

### 2.8 操作稳定性 ✅
Block 操作函数一旦测试通过，不得为了适配 Container 需求而修改。Container 层只能包装，不能改变 Block 操作本身。

### 2.9 格式化不改结构 ❓
格式化命令（粗体、颜色等）只改内容，不改容器结构。改结构的只有显式操作（Tab 嵌套、turnInto）。
> 问题：这个原则是否过于细节，属于 mirro-desktop 实现层面？还是足够通用？

## 三、抽象与复用原则

### 3.1 相似即可抽象 ✅
两个以上功能有相同结构或行为模式时，提取共同抽象。

### 3.2 先具体后抽象 ✅
先用具体实现验证模式稳定，再提取抽象层。不提前抽象。

### 3.3 抽象要有收益 ✅
抽象的目的是减少重复、统一行为、降低认知负担。不为抽象而抽象。

### 3.4 已识别的抽象机会 ✅
- **EmbedBlock 模式**：Video、Tweet、NoteLink 等外部资源嵌入，结构相似
- **Converter 模式**：所有 Block 的 Atom ↔ Editor 双向转换，接口可统一
- **Container 共享行为**：折叠/展开、子节点管理、嵌套——可抽象基类
- **Controller 编排模式**：主进程 Controller 协调子系统，不含业务逻辑
- **Store 模式**：数据持久化抽象为 Store 类，可替换底层实现
- **Pipeline 模式**：提取管线 DOM → ExtractedBlock → Atom → Editor，多阶段标准化

## 四、模块组织原则

### 4.1 一类节点 = 一个自包含模块 ✅
View、Commands、Plugin 聚合在一起，不分散到全局文件。

### 4.2 模块导出契约 ✅
每个模块通过 index.ts 导出标准接口（view、commands、plugin）。

### 4.3 注册即生效 ❓
新增节点只需实现模块 + 注册，不需要修改框架代码。
> 问题：mirro-desktop 当前注册分散在 7-8 个文件，重构时是否改为声明式注册？

### 4.4 命令归属层级 ✅
- 节点专有命令 → 该节点模块
- 跨节点共享命令 → shared 或全局
- Block 转换命令 → 统一转换层

### 4.5 目录演进规则 ❓
Container 立即组织为子目录；Block 从扁平文件开始，膨胀时才建目录。
> 问题：这是否太具体？重构时可能完全不同的目录策略。

## 五、数据层原则

### 5.1 Atom 是框架无关的最小数据单元 ✅
存储层不依赖任何编辑器框架。换引擎时 Atom 不变。

### 5.2 双向转换器注册表 ✅
Atom ↔ Editor 转换通过注册表模式，O(1) 查找，可扩展。

### 5.3 提取管线标准化 ✅
内容提取遵循固定管线：原始内容 → 结构化中间格式 → Atom → Editor。每个阶段职责明确。

### 5.4 类型命名约定 ❓
Atom 用 kebab-case，编辑器节点用 camelCase，转换器负责翻译。
> 问题：重构时是否统一命名？还是保持分层各自的约定？

## 六、UI / 交互原则

### 6.1 View 拥有自己的 Toolbar ✅
每个 View 内部管理自己的 Toolbar，不由外部 Shell 统一管理。零 IPC 开销，直接 API 调用。

### 6.2 视图焦点驱动布局切换 ✅
用户在某个 View 中操作时，UI 自动切换到该 View 的主布局。用户焦点即意图。

### 6.3 布局模式枚举化 ❓
所有视图排列方式预定义为枚举，不动态构造。
> 问题：重构时是否需要更灵活的布局系统？

### 6.4 不用过时数据覆盖用户修正 ✅
始终读取当前版本再修改，不基于旧版本重写。用户手动修正的内容不可被覆盖。

---

## 七、IPC / 进程通信原则

### 7.1 IPC 通道按领域分组 ✅
通道按业务域组织（NOTE_IPC、PDF_IPC 等），使用 `as const` 保证类型安全。

### 7.2 Divider 拖拽用 IPC 驱动 ❓
拖拽交互通过 IPC 事件驱动，不用轮询。
> 问题：这是 Electron 特有的模式，重构时技术栈可能不同。

---

## 八、开发约束

### 8.1 不过度设计 ✅
只实现当前需要的功能，不为假设的未来需求添加复杂度。

### 8.2 分阶段交付 ✅
功能明确分配到 Phase 1/2/3，当前阶段只实现当前阶段的功能。

### 8.3 每次修改后回归测试 ✅
对核心操作层的修改，必须运行完整功能测试验证无回归。

### 8.4 继承检查 ✅
修改 Block 基类操作后，确认所有 Container 中的 Block 仍然正常。

### 8.5 容器嵌套测试 ✅
新增或修改 Container 时，测试容器互相嵌套的场景。

## 九、Electron / ProseMirror 特有原则（可能不带入重构）

### 9.1 Named imports for Tiptap ❌
Tiptap v3 包必须使用 named import。（框架特有）

### 9.2 不用 addKeyboardShortcuts ❌
复杂快捷键用 ProseMirror plugin handleKeyDown。（框架特有）

### 9.3 不用 view.dom.blur() ❌
用 CSS 隐藏光标而非 blur。（实现细节）

### 9.4 WebContentsView Z-Order ❌
Electron 视图叠加顺序。（Electron 特有）

### 9.5 Electron Menu vs Editor 快捷键冲突 ❌
registerAccelerator: false + before-input-event。（Electron 特有）

### 9.6 No BubbleMenu without EditorProvider ❌
Tiptap BubbleMenu 的已知限制。（框架特有）
